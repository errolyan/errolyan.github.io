# 第181期 何时使用 Claude Skills（LLM的工具箱） 与 MCPs（LLM的USB接口）？


Skills（技能）可教会 Claude 如何更好地完成任务，而 MCPs（模型上下文协议）则能让 Claude 接触到新的内容。下文将介绍二者各自的适用场景。


最近，Anthropic 推出了 Claude Skills，我立刻产生了一个疑问：它与模型上下文协议（MCPs）有何不同？二者都能通过 Claude Code 和 Claude Desktop 融入你的工作流程，但起初我并不清楚它们之间的区别。

在深入研究二者之后，我发现产生这种困惑是情有可原的，但答案其实比想象中更简单——它们要解决的是本质上不同的问题。

##  Claude Skills 究竟是什么
当 Anthropic 推出 Claude Skills 时，我最初以为它只是另一种提供上下文的方式。但实际上，它的定位要具体得多。

Claude Skills 是包含指令、脚本和资源的文件夹，当这些内容与你的任务相关时，Claude 会自动加载。可以将其视为“专业知识包”——你为 Claude 整合的专项知识，供其调用。

正如 Anthropic 所描述的，若将 MCPs 比作 USB-C 接口（一种连接外部数据源的标准化方式），那么 Claude Skills 就更像是你交给工匠的专用工具箱。工匠（即 Claude）会在需要时自动取用合适的工具。

Claude Skills 有一个本质上的不同之处：它包含可执行代码，而非仅仅是提示词。当你创建一个用于处理 Excel 电子表格的 Skill 时，Claude 不仅能理解 Excel 格式，还能运行代码来持续、可靠地修改你的文件。对于需要精准操作的任务，无需再靠猜测。

Claude Skills 还具备可组合性。当你提出需求时，Claude 会扫描可用的技能，并协调它们共同发挥作用。例如，你的“品牌设计指南”Skill 和“制作演示文稿”Skill 能自动协同工作，无需你手动选择，一切都会自动完成。

##  MCPs 的适用场景
模型上下文协议（MCP）能让 Claude 访问外部系统和实时数据，它就像是 Claude 与你的集成开发环境（IDE）、Slack、GitHub、Anki 或任何 Claude 原生环境之外的系统之间的桥梁。

Anki MCP 服务器就是一个实际案例。它并非教会 Claude 如何更好地完成某项任务，而是让 Claude 能够查看你的卡片组、抽认卡和笔记。Claude 可以读取这些数据，并提出改进建议。这体现的是“访问能力”，而非“专项技能”。

MCPs 需要独立的服务器基础设施支持。通常情况下，你需要在本地或某个服务中运行 MCP 代理，然后让 Claude Desktop 与之连接。相较于 Claude Skills，MCPs 的设置更为复杂，但它的优势在于能与外部系统集成，而这是仅靠 Skills 无法实现的。

##  实际应用中的区别
下面我将通过真实场景来详细解读二者的区别：

### 何时使用 Claude Skills
当你希望 Claude 处理可重复的工作流程，且需要保证高度一致性时，可使用 Claude Skills。例如，你希望自己创建的每一个电子表格都遵循特定的公式结构、配色方案和版式，以符合公司的标准。这时，你可以创建一个 Skill，用可执行代码将这些规则编码其中，Claude 会自动加载该 Skill。任务就此完成，无需在每次对话中进行配置，只要任务相关，它就能自动发挥作用。

### 何时使用 MCPs
当你需要 Claude 完成一些它在独立环境中无法完成的任务时，就需要用到 MCPs。比如，你在使用 Anki 程序时，希望 Claude 分析你的抽认卡，包括那些你并未明确展示给它的数据。这时，你可以搭建 Anki MCP，这样 Claude 就能立即查看你的卡片组和标记的卡片，并能自动针对所有数据提出改进建议。

### 何时同时使用二者
你可以先搭建 Anki MCP，让 Claude 能够读取所有卡片（实现“访问”），然后再使用一个 Skill，确保 Claude 在提出修改建议或创建新卡片时，都能遵循你设定的特定格式规则（实现“专项优化”）。

二者最核心的区别在于：MCPs 侧重于“访问能力”，而 Claude Skills 侧重于“专项优化”。MCPs 回答的是“Claude 能看到什么？”，而 Skills 回答的是“Claude 如何能把这件事做得更好？”

##  如何创建自己的 Claude Skill
Claude Skills 最让我印象深刻的一点是，它极大地降低了创建门槛。实际上，它有一个“技能创建器”（skill-creator）Skill——这是一个元技能（meta-skill），专门用于帮助你创建其他技能。你无需手动创建文件夹结构或编写 YAML 文件，只需描述你想要的功能，Claude 就会通过交互式引导，协助你完成创建。

例如，如果你想创建一个用于处理 PDF 模板的 Claude Skill，只需开启一次对话，请求 Claude 帮你创建该 Skill，之后它会按以下步骤操作：
1. 询问你的工作流程；
2. 生成合适的文件夹结构；
3. 格式化 SKILL.md 文档文件；
4. 打包你所需的所有资源。

整个过程无需手动编辑文件。相比之下，搭建 MCP 服务器需要运行进程、配置连接详情并处理错误日志。Claude Skills 的设计初衷就是为了方便使用，而 MCPs 则需要更多关于基础设施的考量。

尽管如此，当你需要访问外部系统时，在 MCPs 基础设施上的投入就会显现出价值。但如果是为了让 Claude 的工作方式更具针对性——比如优化它的输出格式、适配你的特定领域模式或符合你所在组织的标准——那么 Claude Skills 的使用门槛更低，操作更便捷。

##  核心区别总结
在实际使用过二者之后，我发现最显著的区别在于：Claude Skills 能让你将自己的知识和工作流程融入 Claude 的行为模式中，而 MCPs 则能让 Claude 突破自身视野的局限，访问更多外部内容。

如果你的问题属于以下类型，那么 Claude Skills 就是你的解决方案：
- “Claude 不了解我们的内部规范”
- “我们希望所有生成的文档都保持一致性”
- “我们需要 Claude 每次都严格遵循这个特定的工作流程”

如果你的问题属于以下类型，那么 MCPs 则是更合适的选择：
- “Claude 需要访问我们的实时数据”
- “我们希望 Claude 能在 IDE 内部工作，无需复制粘贴”
- “Claude 需要与外部系统进行实时集成”

二者并非互斥关系，而是互补关系，你可以在同一个工作流程中同时使用它们。

##  如何选择：你需要哪一个？
不妨问问自己：我是想让 Claude 的行为更具针对性，还是需要扩展 Claude 的访问范围？或者两者都需要？

答案就此揭晓。

如果你刚开始尝试扩展 Claude 的功能，建议从 Claude Skills 入手。它的使用门槛更低，能快速看到效果。你可以围绕自己经常重复的工作内容（如文档格式化、代码结构规范、输出一致性保障等）创建一个 Skill，并借助“技能创建器”来完成操作。

当你熟悉了 Skills 的使用方法后，如果你发现自己需要“让 Claude 实时查看我的数据”，这时就可以考虑使用 MCPs 了。到那时，你已经清楚二者的区别，也能更好地理解这两种工具的实际用途。
